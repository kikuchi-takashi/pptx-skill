# シグナル判定基準

`scripts/scan.py` が既定で判定するシグナル一覧。3系統すべてを毎回判定するが、
複数が重なった銘柄ほど「候補銘柄」セクション内で優先度（表示順）を上げる
（シグナル検知数が多い順にソートするだけで、複雑な重み付けスコアリングは行わない）。

## 1. テクニカル

| シグナル | 判定 | 一言の意味 |
|---|---|---|
| 移動平均クロス | SMA25とSMA75の差分の符号が直近3日以内に反転（ゴールデンクロス=上抜け／デッドクロス=下抜け） | トレンド転換の兆候 |
| RSI過熱 | RSI(14) が70以上（買われすぎ）または30以下（売られすぎ） | 短期的な行き過ぎの目安 |
| ボリンジャーバンド・ブレイク | 終値がSMA20±2σのバンドを上抜け／下抜け | 通常レンジからの逸脱 |

RSI・ボリンジャーバンドの計算式は `market-data-report/scripts/make_report.py` の
`compute_indicators()` と同じ考え方（14日RSI、SMA20±2σ）を踏襲しているが、
本スキル (`scan.py`) は独立に再実装している。両スキルは出力形態（単体の
チャート付きレポート vs 複数銘柄横断のスキャン）が異なり、無理に共通モジュール化
すると結合度が上がると判断したため。`portfolio-risk` が価格取得ロジックを独自に
持っている前例と同じ考え方（詳細は `portfolio-risk/SKILL.md` 冒頭を参照）。

## 2. ボラティリティ・出来高の異常値（情報の新鮮度シグナル）

| シグナル | 判定 | 一言の意味 |
|---|---|---|
| 出来高急増 | 直近出来高が過去20日平均の2.0倍以上 | 商いの急な変化＝何らかの材料が出ている可能性 |
| ボラティリティ急上昇 | 直近5日の日次リターン標準偏差が直近60日平均の1.5倍以上 | 値動きの荒さが急に増している |

いずれも「なぜ動いたか」の理由までは特定しない（それはニュース等別途確認が必要）。
あくまで「直近の値動き・出来高に通常と異なる点がある」という新鮮度シグナル。

## 3. ファンダメンタル

| 項目 | 取得方法 | 扱い |
|---|---|---|
| PER（実績・予想） | `yfinance` の `Ticker.info` の `trailingPE`/`forwardPE` | 候補ごとの参考情報として表示。単体では「割高/割安」の断定はしない |
| 直近決算発表日 | `yfinance` の `Ticker.get_earnings_dates()` | 次回発表予定日までの日数が14日以内なら「決算発表が近い」シグナルとして検知（イベントリスク・情報の新鮮度の目安） |

`yfinance` の `info`/`get_earnings_dates()` は銘柄・タイミングによって値が
欠けたり、例外を投げたりすることがある（Yahoo Finance側の仕様変更・レート制限等）。
取得できない項目は握り潰さず「データなし」と明記し、他の項目・他銘柄の処理は
継続する（1項目の欠落でスキャン全体を止めない）。

## 閾値の一覧（`scan.py` 冒頭の定数）

- `RSI_OVERBOUGHT = 70.0` / `RSI_OVERSOLD = 30.0`
- `MA_CROSS_LOOKBACK_DAYS = 3`
- `VOLUME_SPIKE_RATIO = 2.0` / `VOLUME_BASELINE_WINDOW = 20`
- `VOL_SPIKE_RATIO = 1.5` / `VOL_SHORT_WINDOW = 5` / `VOL_LONG_WINDOW = 60`
- `EARNINGS_SOON_DAYS = 14`

これらは経験則的な既定値であり、最適化されたものではない。ユーザーから
「もっと厳しく/緩く」等の要望があれば、スクリプト冒頭の定数を調整する
（CLIオプション化はしていない。頻繁に変える性質のものではないため、まずは
定数直書きに留めている）。

## レポートに書く際の注意

- シグナルは「事実の記述＋兆候の示唆」に留める。「買われすぎ→今売るべき」のような
  断定的な投資判断・売買推奨は書かない。「〜の兆候が見られる」というトーンで統一する。
- 候補提示までがこのスキルの範囲。実際に発注したい場合は「`order-ticket-draft`
  スキルで下書きを作成できます」と案内するだけに留め、自動で発注や下書き生成を
  行わない。
