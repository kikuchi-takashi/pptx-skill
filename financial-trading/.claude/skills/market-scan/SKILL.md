# 取引候補スキャン・スキル

ウォッチリスト（保有していない監視銘柄）と保有ポートフォリオ（現在保有している
銘柄）の両方を対象に、テクニカル・出来高/ボラティリティ・ファンダメンタルの3系統の
シグナルを横断的にチェックし、複数シグナルが重なった銘柄を優先して見せる
Markdownレポートを作成する。

**このスキルはユーザーが「今日のスキャンして」のように自分から都度話しかけたときに
だけ動く。** cron・スケジューラ等による自動・無人実行の仕組みは一切実装していない
し、実装しない（定期実行の仕組み自体が今回のスコープ外）。

**免責事項**: 本スキルが提示する候補・シグナルは情報提供のみを目的とし、投資助言
ではない。「〜の兆候が見られる」という記述はあくまで参考情報であり、断定的な
「買い/売り推奨」ではない。投資判断・発注の実行は必ず自己責任で行うこと。

---

## このスキルが対象とすること／しないこと

- 対象: ウォッチリスト銘柄・保有ポートフォリオ銘柄を対象にした、テクニカル
  （移動平均クロス・RSI過熱・ボリンジャーバンド・ブレイク）、出来高/ボラティリティ
  異常値（出来高急増・ボラティリティ急上昇）、ファンダメンタル（PER・直近決算発表日）
  の3系統のシグナル判定と、候補銘柄をまとめたMarkdownレポートの作成。
- 対象外: **発注・自動売買は一切行わない。** 候補の提示までがこのスキルの範囲。
  実際に発注したい場合は「`order-ticket-draft` スキルで下書きを作成できます」と
  案内するだけに留め、本スキルが自動で `order-ticket-draft` を呼び出したり注文
  チケットを生成したりすることはない。
- 対象外: 定期実行・無人実行（cron的なスケジュール実行の仕組みは持たない。必ず
  ユーザーからの都度のリクエストで発火する）。
- 対象外: 過去データに対する売買ルールの勝率検証（バックテスト）。
- 対象外: 詳細なリスク指標算出（ボラティリティ・VaR・集中度・相関等は
  `portfolio-risk` スキールの領分）。本スキルが扱うボラティリティ・出来高チェックは
  あくまで「直近の値動きに通常と異なる点があるか」という新鮮度シグナルに限る。

## セットアップ

```bash
pip3 install yfinance pandas numpy
python3 -c "import yfinance, pandas, numpy; print('deps OK')"
```

データソースは他の2スキル（`market-data-report`, `portfolio-risk`）と同じく
[yfinance](https://github.com/ranaroussi/yfinance)（APIキー不要・無料）。
ファンダメンタル情報（`Ticker.info`, `Ticker.get_earnings_dates()`）は特に
Yahoo Finance側の仕様変更・レート制限の影響を受けやすく、取得できないことが
珍しくない。取得失敗時は該当項目を「データなし」として明記し、スキャン自体は
継続する（クラッシュさせない設計。詳細は `references/SIGNALS.md` を参照）。

---

## 入力: ウォッチリストCSV・保有ポジションCSV（少なくとも一方）

### ウォッチリストCSV（本スキル専用の新形式）

```csv
ticker
7203.T
9984.T
AAPL
MSFT
BTC-USD
```

`references/watchlist_template.csv` にサンプルを用意してある。ユーザーが
まだ持っていなければこのテンプレートをコピーして使ってよいか確認する。

### 保有ポジションCSV（`portfolio-risk` と同じ形式を再利用・新形式は作らない）

```csv
ticker,quantity,cost_basis
7203.T,100,2100
AAPL,10,180.5
```

**このスキル独自のポートフォリオCSV形式は新設しない。** `portfolio-risk` スキルが
使っている `ticker,quantity,cost_basis` 形式のCSVをそのまま `--positions` に渡せる。
本スキルは `quantity`/`cost_basis` は使わず、対象銘柄（ticker列）の抽出だけに使う
（詳細な時価評価・リスク評価は `portfolio-risk` スキルに任せる）。ユーザーが
既に `portfolio-risk` 用のCSVを持っていれば、それをそのまま指定すればよい。

`--watchlist` と `--positions` は両方指定してもよいし、どちらか一方だけでもよい
（少なくとも一方は必須）。両方に登場するティッカーは重複排除され、レポート上に
「対象: watchlist・portfolio」のように出所が明記される。

## 実行フロー

### Step 1: 対象を確認する
ウォッチリストCSVのパス、保有ポジションCSVのパス（`portfolio-risk` 用のものを
流用可）を確認する。ユーザーが「ウォッチリストだけ見て」「保有銘柄だけスキャンして」
と言えば片方だけを渡せばよい。どちらも未指定・ファイルがなければユーザーに
準備を促す（`references/watchlist_template.csv` を提示してもよい）。

### Step 2: スキャンを実行する
```bash
python3 scripts/scan.py --watchlist ./watchlist.csv --positions ./positions.csv \
  --period 6mo --out ./output/market_scan_report.md
```
判定基準・閾値の定義は `references/SIGNALS.md` を参照。個別銘柄の取得失敗
（ティッカー誤り・レート制限等）はその銘柄をスキャン対象から除外した上で
「データ取得上の注意」としてレポートに明記され、スキャン全体は中断しない。

### Step 3: レポートを読んで検証する
生成された Markdown レポートを **Read ツールで確認する**。
チェック項目:
- 「候補銘柄」セクションのシグナルが `references/SIGNALS.md` の閾値と整合しているか
- 断定的な「買い/売り推奨」表現になっていないか（「〜の兆候が見られる」トーンに
  統一されているか）
- ファンダメンタルのデータ欠落が「データなし」として隠さず明記されているか
- 保有銘柄とウォッチリスト銘柄で出所が正しく区別されているか
- レポート内に「発注は行わない」旨・`order-ticket-draft` への案内文言が
  含まれているか（含まれない場合は報告時に口頭で補う）

### Step 4: 報告
候補銘柄とそのシグナル・理由をユーザーに提示する。複数シグナルが重なった銘柄は
優先度を上げて伝えてよいが、あくまで「〜の兆候が見られる」というトーンに
留める。実際に発注したい銘柄があれば、**「`order-ticket-draft` スキルで下書きを
作成できます」と案内するだけに留め、自動で発注や下書き生成を行わない。**
