# 実発注・自動売買（ドライラン限定）スキル

## 安全方針（最重要・絶対要件——このスキルの存在意義そのもの）

**本スキルは常にドライランである。実際の注文送信（証券会社・取引所APIへの発注実行）は
一切行わない。** このスキルが行ってよいのは、注文内容の整理・妥当性検証・注文チケット
（テキスト/CSV）の生成までであり、それ以上ではない。

- スキル自身が発注APIを呼び出す、ブラウザ操作で取引画面の「注文」ボタンを押す、
  自動売買ループを常駐実行する——これらは**すべて禁止**。資金が動く操作を
  スキルが自律的に実行することは絶対にない。
- 生成した注文チケットは、**人間が内容を確認し、証券会社の取引画面に自分の手で
  入力するか、証券会社が対応していれば注文用CSVとしてインポートする**ための
  下書きに過ぎない。「送信」に相当する操作はユーザー自身が行う。
- `scripts/build_ticket.py` は注文チケットの生成・検証のみを行うスクリプトであり、
  ネットワーク経由での発注APIコール・ブラウザ自動操作コードを一切含まない
  （含めてはならない）。このスキルを拡張・修正する際も、この制約を絶対に破らない。
- レポート・チケットの冒頭には毎回「⚠️ ドライラン — このチケットは下書きです。
  実際の注文はご自身で証券会社の画面から行ってください」という警告を明記する。

ユーザーが「自動で発注しておいて」「このまま送信して」のように実送信を求めてきた場合、
**実行せず、その操作はできない旨と、生成したチケットを使って手動で発注する手順を
案内する。**

**免責事項**: 本スキルが生成する注文案・検証結果は情報提供のみを目的とし、投資助言
ではない。投資判断・発注の実行は必ず自己責任で行うこと。

---

## このスキルが対象とすること／しないこと

- 対象: 発注内容の整理（銘柄・数量・売買区分・成行/指値・価格条件・有効期限）、
  ポジションサイズ上限やリスク上限との整合性チェック、注文チケット（テキスト/CSV）の
  生成。
- 対象外: 実際の発注送信、自動売買の常駐実行、バックテスト・戦略検証。

自体はネットワークアクセスを行わない）。

## 実行フロー

### Step 1: 発注内容をヒアリングする
以下を確認する（不明な項目は必ず聞く。推測で埋めない）:
- 銘柄（ティッカー）
- 売買区分（買い/売り、新規/返済）
- 数量
- 注文種別（成行/指値/逆指値）と価格条件
- 有効期限（当日限り/期間指定等）
- 口座種別（現物/信用等、該当する場合）

### Step 2: 整合性チェック
`portfolio-risk` スキルの出力やユーザー申告の資産規模がある場合、以下を確認し、
基準を超える場合はチケットに警告として明記する（発注自体は止めない——判断は
ユーザーに委ねる）。いずれも `scripts/build_ticket.py` の対応するオプションを渡せば
チケット本文に警告として自動反映される:
- 1銘柄あたりの発注金額がポートフォリオ総額の一定割合（既定20%、ユーザー指定があれば
  それに従う）を超えていないか（`--portfolio-value` [`--max-position-pct`]）
- 保有数量を超える売り注文になっていないか（申告があれば。`--holding-quantity`）
- 指値価格が直近価格（`--reference-price`）から既定5%以上乖離していないか（誤入力の
  可能性）

### Step 3: 注文チケットを生成する
```bash
python3 scripts/build_ticket.py --ticker 7203.T --side buy --quantity 100 \
  --order-type limit --price 2150 --tif day --portfolio-value 3000000 \
  --reference-price 2100 --out ./output/order_ticket.txt
```
売り注文で保有数量の超過チェックも行いたい場合は `--holding-quantity <保有数量>` を
追加で渡す。
`--out` の拡張子が `.csv` の場合、CSV本体と、同じ内容を人が読める形にした確認用
テキスト（`<ファイル名>_readme.txt`）の2ファイルを出力する。拡張子が `.csv` 以外
（例: `.txt`）の場合はテキストのみを出力する（CSVは生成されない）。両方の形式を
別ファイルとして欲しい場合は、拡張子を変えて2回実行する。中身は「証券会社の取引
画面に転記するための下書き」であり、送信可能な形式（APIリクエスト等）では出力
しない。

### Step 4: 内容を読んで確認する
生成されたチケットを **Read ツールで確認する**。
チェック項目:
- ドライラン警告が冒頭に明記されているか
- 数量・価格・売買区分に誤記がないか（ヒアリング内容と一致しているか）
- 整合性チェックの警告（あれば）が正しく反映されているか
- 誤って「送信済み」「発注完了」等の実行を示唆する文言が入っていないか
  （本スキルは常に「下書き」「要確認」のトーンで統一する）

### Step 5: 報告
チケットの内容をユーザーに提示し、**「この内容を証券会社の取引画面でご自身が
確認の上、手動で発注してください」と明示的に案内する。** スキル側で送信は行わない旨も
併せて伝える。
