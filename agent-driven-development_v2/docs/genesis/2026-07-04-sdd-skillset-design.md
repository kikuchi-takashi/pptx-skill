# 設計書：spec-driven development スキルセット（agent-driven-development_v2）

日付: 2026-07-04
ステータス: ユーザーレビュー待ち

## 目的

superpowers（プロセス規律・サブエージェント駆動・検証文化）と spec-kit（フェーズ構造・成果物設計・整合性検証）を参考に、**汎用の仕様駆動開発ワークフロー**を新規スキルセットとして `agent-driven-development_v2` に構築する。既存 v1（superpowers コピー）には手を加えない。

## 決定事項（要件ヒアリング結果）

| 項目 | 決定 |
|---|---|
| 用途 | 汎用（どの開発プロジェクトでも使える） |
| 起動方式 | ハイブリッド（オーケストレーター自動発火＋各フェーズ単体呼び出し可） |
| フェーズ構成 | フルセット8フェーズ＋小規模バイパス |
| 言語 | スキル本文・成果物とも日本語。description に日本語トリガー例 |
| 実装方式 | サブエージェント駆動を基本、小タスクは本体直接（自動選択） |
| アーキテクチャ | 案A：オーケストレーター＋状態ファイル（specs/NNN-機能名/state.md） |

## ディレクトリ構成

```
agent-driven-development_v2/
├── .claude/skills/
│   ├── sdd/                  # オーケストレーター（入口）
│   ├── sdd-constitution/     # フェーズ1: プロジェクト原則
│   ├── sdd-specify/          # フェーズ2: 仕様書
│   ├── sdd-clarify/          # フェーズ3: 曖昧箇所解消
│   ├── sdd-plan/             # フェーズ4: 技術計画
│   ├── sdd-tasks/            # フェーズ5: タスク分解
│   ├── sdd-analyze/          # フェーズ6: 整合性検証
│   ├── sdd-implement/        # フェーズ7: 実装
│   │   └── implementer-prompt.md   # サブエージェント用プロンプト雛形
│   └── sdd-verify/           # フェーズ8: 検証・完了
├── docs/constitution.md      # sdd-constitution が生成（プロジェクトごと初回のみ）
└── specs/
    └── NNN-機能名/            # 機能ごと（NNN は 001 からの連番）
        ├── state.md          # 現在フェーズ・承認履歴・次アクション
        ├── spec.md           # 仕様（ユーザーストーリー＋受け入れ基準）
        ├── clarifications.md # Q&A 記録
        ├── plan.md           # 技術計画
        ├── tasks.md          # タスク分解
        └── spec-lite.md      # 小規模バイパス時のみ（spec/plan/tasks を1枚に）
```

各スキルは `SKILL.md` 1ファイル（＋sdd-implement のみ implementer-prompt.md を持つ）。SKILL.md は各500行以内に収める。

## スキル仕様

### sdd（オーケストレーター）

- **description（発火トリガー）**: 開発依頼全般。「〜を作って」「機能を追加して」「〜を実装したい」「開発を再開して」「続きから」
- **起動時の動作**:
  1. `specs/*/state.md` を走査。進行中（フェーズ未完了）の機能があれば一覧を提示し「続きから再開するか」を確認
  2. 新規なら `docs/constitution.md` の有無を確認。なければ sdd-constitution を先に実行
  3. 依頼の規模を判定（下記バイパス基準）→ フルルート or 短縮ルート
  4. `specs/NNN-機能名/` を採番・作成し、state.md を初期化
  5. 現在フェーズに対応するスキルを Skill ツールで呼び出す。以降フェーズ完了→承認→次フェーズを繰り返す
- **ゲート規則（HARD GATE）**:
  - フェーズ完了時は必ず成果物の要約を提示してユーザー承認を取る
  - 承認は state.md に「日時＋承認内容」で記録
  - 承認記録がないフェーズをスキップして先のスキルを呼ぶことを禁止
  - 例外: clarify は「曖昧箇所ゼロ」の場合のみ自動通過可（通過した事実は state.md に記録）
- **小規模バイパス基準**（すべて満たす場合のみ）:
  - 変更が1〜2ファイルに収まる見込み
  - 既存の spec.md の受け入れ基準に影響しない
  - 新しい依存関係・データモデル変更を伴わない
  - → specify/plan/tasks を `spec-lite.md`（変更内容・受け入れ基準・検証コマンドの3節）1枚に畳み、implement → verify へ直行。バイパス採用は state.md に理由付きで記録

### state.md フォーマット

```markdown
# 001-user-auth 進行状態
- 機能名: ユーザー認証
- ルート: フル | バイパス
- 現在フェーズ: specify | clarify | plan | tasks | analyze | implement | verify | 完了
## 承認履歴
- 2026-07-04 14:00 specify 承認（ユーザー）
## フェーズ内進捗（implement 中のみ）
- [x] T1: モデル定義（subagent / 直接）
- [ ] T2: ...
## 差し戻し履歴
- （日時・フェーズ・理由）
```

### sdd-constitution

- **トリガー**: 「プロジェクト原則を決めて」／sdd からの初回呼び出し
- 技術スタック方針・品質基準（テスト必須度、カバレッジ観）・禁止事項・レビュー方針を対話で確定し `docs/constitution.md` に書く。既存プロジェクトではコード規約・CLAUDE.md を先に読んで矛盾しない案を出す
- 以後の plan / analyze / implement はこのファイルを必ず参照する

### sdd-specify

- **トリガー**: 「仕様書を書いて」「要件をまとめて」
- **WHAT/WHY のみ。HOW（技術選定・実装方法）の混入を禁止**（本文に禁止例を列挙）
- 出力: ユーザーストーリー（〜として、〜したい、なぜなら〜）＋受け入れ基準（Given/When/Then）＋スコープ外リスト
- 判断できない点は `[要確認: 〜]` マーカーを本文に残す（clarify の入力になる）

### sdd-clarify

- **トリガー**: 「仕様の曖昧なところを詰めて」
- spec.md の `[要確認]` と、独自に検出した曖昧点を **1問ずつ・選択式優先** で質問（superpowers brainstorming 流）。上限5問、超える場合は影響の大きい順
- Q&A を clarifications.md に記録し、spec.md 本文へ反映して `[要確認]` を消す。全部消えたら完了

### sdd-plan

- **トリガー**: 「技術計画を立てて」「アーキテクチャを設計して」
- 入力: spec.md（承認済みであること）＋constitution.md
- 出力: 技術選定（理由付き）・コンポーネント分割・データモデル・エラー処理方針・テスト戦略
- **constitution 準拠チェック節を必須**とし、逸脱がある場合は理由と代替案を明記

### sdd-tasks

- **トリガー**: 「タスクに分解して」
- plan.md を実装タスク列に分解。各タスクに必須4項目: ①変更対象ファイル ②完了条件 ③検証コマンド（実行可能なもの） ④依存タスク
- 依存のないタスク同士に `[P]`（並列可能）マーク
- 1タスクの粒度は「サブエージェントが1回の派遣で完了できる大きさ」（目安: 関連ファイル5個以内）

### sdd-analyze

- **トリガー**: 「整合性をチェックして」
- **read-only**（成果物の書き換えはせず報告のみ）
- 検証項目: ①受け入れ基準→タスクのカバレッジ表（漏れ検出） ②spec↔plan の矛盾 ③constitution 違反 ④検証コマンドの実行可能性
- 重大問題があれば該当フェーズへの差し戻しを提案（差し戻しは state.md に記録）

### sdd-implement

- **トリガー**: 「実装して」（tasks.md 承認済みが前提。未承認なら sdd に差し戻す）
- **実行方式の自動選択**: タスクの変更対象が1〜2ファイル → 本体セッションで直接。それ以上 → サブエージェント派遣。判断は tasks.md の記載に基づき、選択結果を state.md に記録
- **サブエージェントに渡すもの**: implementer-prompt.md（TDD 強制: 失敗するテストを先に書く→最小実装→リファクタ）＋該当タスク定義＋spec.md の関連受け入れ基準＋constitution.md。**plan 全文や他タスクは渡さない**（コンテキスト隔離）
- `[P]` タスクは同時派遣可（ただし同一ファイルを触るタスクの同時派遣は禁止）
- タスク完了ごとに本体が検証コマンドを実行して確認し、state.md のチェックボックスを更新
- **仕様矛盾を発見したら**: 実装を止めて報告。spec.md を差し戻し修正（承認必要）→ 影響タスクを更新してから再開。黙って仕様と違うものを作らない

### sdd-verify

- **トリガー**: 「検証して」「完了確認して」
- spec.md の全受け入れ基準を1件ずつ実際に実行し、結果（コマンド＋出力）を証拠として記録
- **証拠なしに「完了」と言うことを禁止**（superpowers verification-before-completion 流）
- 全件パス後: state.md を「完了」にし、統合方法（コミット／PR／そのまま）の選択肢を提示

## description 設計方針

- 全スキル: 英語の状況記述は使わず**日本語で状況＋トリガー例**を書く（本文が日本語のため統一）
- 各フェーズスキルの description に「通常は sdd オーケストレーター経由で呼ばれる。単体でも呼び出し可」と明記し、二重発火（sdd と個別スキルの競合）を防ぐ

## エラー処理

- **セッション中断**: sdd が state.md から再開（案Aの核心）。state.md が壊れている場合は成果物の存在から状態を推定し、ユーザーに確認してから復元
- **フェーズ差し戻し**: どのフェーズからでも前フェーズへ戻せる。戻す際は理由を state.md の差し戻し履歴に記録し、下流成果物（例: spec 変更時の plan/tasks）の更新要否を明示
- **サブエージェント失敗**: 2回失敗したタスクは本体セッションで引き取り、systematic-debugging 的に原因を特定してからタスク定義を修正

## テスト計画（スキルセット自体の検証）

1. **発火テスト**: サブエージェントで新規会話をシミュレートし、「TODOアプリ作って」→ sdd 発火、「バグ直して」→ 発火しない（スコープ外）ことを確認
2. **通しドライラン**: サンプル機能（CLI 電卓程度）で全8フェーズを実走。ゲートで止まるか・state.md が正しく更新されるかを確認
3. **再開テスト**: ドライランを plan 承認後に中断 → 新セッションで「続きから」→ tasks から再開できることを確認
4. **バイパステスト**: 小修正依頼で spec-lite ルートに入ることを確認

## スコープ外

- v1（agent-driven-development/.claude）の変更・削除
- SessionStart フック（v1 で権限ブロックされた経緯があるため、v2 では sdd の description 発火のみで運用開始し、必要になったら別途追加）
- CI／GitHub Actions 連携
