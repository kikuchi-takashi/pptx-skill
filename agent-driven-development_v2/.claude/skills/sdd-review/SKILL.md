# sdd-review — 実装レビュー

汎用のコードレビューではなく、**この機能の契約（spec.md・constitution.md・tasks.md）との突き合わせ**を行う。

## モード

- **タスクレビュー**: implement 中、サブエージェント実装タスクの完了ごとに sdd-implement から呼ばれる。対象は当該タスクの diff のみ
- **フィーチャーレビュー**: implement フェーズ完了後の review フェーズ。対象は機能の全変更。タスクレビューでは見えない**タスク間の不整合**（重複実装・命名の食い違い・つなぎ目の欠落）を重点的に見る

## 前提チェック（フィーチャーレビュー時）

state.md の現在フェーズが review でなければ sdd に差し戻す。

## 手順

1. 対象 diff を特定する（git diff。git 管理外なら state.md のフェーズ内進捗からタスクの変更対象ファイルを集める）
2. **仕様準拠**: 受け入れ基準 AC-n と突き合わせ、不足（実装漏れ）・過剰（仕様にない機能）・誤解（違うものを作った）を検出する
3. **constitution 検算**: 全項目について違反の有無を確認する（禁止事項・品質基準）
4. **品質**: エラー処理の妥当性／テストが実質を検証しているか（モックの動作だけを検証していないか・アサーションのないテスト）／同一ロジックの重複
5. レビューはサブエージェントに派遣してよい（実装した本人以外の目を通すため推奨）。実装報告は未検証の主張として扱い、必ず diff で裏取りする。派遣時はモデルを明示する: タスクレビューは diff の規模・リスクに見合うモデル、フィーチャーレビューは最上位モデル
6. **本体が直接実装した部分を自己レビューする場合は死角に注意**——「自分の書いたロジックは正しいはず」の確証バイアスが構造的にかかる。diff を1行ずつ読む・エラー経路と例外の継承関係は特に疑う。可能ならこの部分こそサブエージェントに出す
7. diff だけでは検証できない要件（未変更コードに依存・タスク横断）は、無理に探索を広げず「⚠️ diff からは検証不能」として列挙する。⚠️ の解消は呼び出し側（sdd-implement / sdd）の責務

## レビュアーへの指示で禁止すること

- 「この点は指摘しなくてよい」「軽微扱いにして」と**事前に判定を縛らない**。指摘は挙げさせてから採否を判断する
- 実装者の「あえてこうした」という弁明で重大度を下げない。理由の説明は実装者の自己採点にすぎない——コードそのものの性質で判定する

## 重大度の較正

- **重大（マージ不可）**: 誤動作・データ破壊・受け入れ基準の未達・アサーションのないテスト
- **重要（修正必須）**: このタスクを信頼できない欠陥——壊れやすい挙動・要件の取り違え・ロジックブロックの逐語重複・握りつぶされたエラー
- **軽微**: 磨き。命名の改善・カバレッジの上積み・コメントの充実
- 「なんとなく気になる」を重要に格上げしない。逆に、指摘の多さを恐れて重大を軽微に格下げしない

## レポート書式

```markdown
## レビューレポート（対象: T<n> | 機能全体）
### 良い点
- <具体的に。正確な称賛が残りの指摘への信頼を作る>
### 仕様準拠: ✅ | ❌
- 不足/過剰/誤解: <内容>（<ファイル>:<行>）
- ⚠️ diff からは検証不能: <要件と、呼び出し側が確認すべきこと>
### 指摘
- 重大（マージ不可）: / 重要（修正必須）: / 軽微:
### 判定: 通過 | implement へ差し戻し
```

すべての指摘に ファイル:行 を付ける。付けられない指摘は根拠が曖昧——diff を読み直す。

## 差し戻し規則

- 重大・重要が1件でもあれば implement へ差し戻す。修正後に**必ず再レビュー**する（修正が新しい問題を持ち込むことは珍しくない）
- state.md の差し戻し履歴に記録するのは**フィーチャーレビューの差し戻しのみ**。タスクレビューの指摘はフェーズ内の修正ループであり、フェーズ差し戻しではない（フェーズ内進捗欄で扱う）
- 軽微は記録のみで通過可（verify の完了報告に一覧を添える）。ただし記録した軽微をどこにも出さないのは黙殺と同じ——フィーチャーレビュー時に一覧を再掲して採否を判断する
- 指摘ゼロでも「何を確認したか」の根拠を書く。「問題なし」だけの報告は禁止
