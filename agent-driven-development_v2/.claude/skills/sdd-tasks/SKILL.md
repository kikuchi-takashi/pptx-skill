# sdd-tasks — タスク分解

## 出力フォーマット（specs/NNN-機能名/tasks.md）

このスキルと同じディレクトリの `template.md` をコピーして埋める。テンプレート内のコメントが記入ガイド。
タスクは `## T<n>: <タスク名> [P]` 見出し＋4項目（変更対象・完了条件・検証コマンド・依存）で書く。

## タスク境界の基準

タスクとは「**独立してテストでき、レビュアーが隣のタスクを承認したままこれだけを差し戻せる最小単位**」。
- 準備作業（セットアップ・設定・雛形作成・ドキュメント）は、それを必要とする成果物のタスクに畳み込む。単独タスクにしない
- 分割するのは「レビュアーが片方だけを合格にできる」境目だけ

## 粒度の良い例・悪い例

| 悪い | 何が問題か | 直し方 |
|---|---|---|
| T1: プロジェクト初期化 / T2: 設定ファイル作成 / T3: モデル実装 | T1・T2 は単独で合否判定できない | T1〜T3 を「モデル実装（初期化・設定込み）」の1タスクに畳む |
| T1: アプリ全体を実装 | 1回の派遣で完了できない。失敗時に全部やり直し | 機能単位（モデル/CLI/永続化 等）に分割 |
| T1: テストを書く / T2: 実装する | RED→GREEN が2派遣に割れ、T1 単独では合格にできない | テスト＋実装を1タスクに |
| T4: リファクタリング（全体） | 完了条件が書けない | 対象と観測可能な完了条件を特定できないなら削除 |

## 規則

（4項目必須・[P] の付け方・テスト＋実装同一タスクの規則はテンプレートのコメントに記載——二重管理しない）

- 粒度: サブエージェントが1回の派遣で完了できる大きさ（目安: 関連ファイル5個以内）。超えるなら分割
- すべての受け入れ基準 AC-n がいずれかのタスクの完了条件に現れること（漏れは analyze で検出されるが、ここで防ぐ）
- タスク間で受け渡すインターフェース（関数名・引数と戻りの型）は、**使う側ではなく作る側のタスクの完了条件に明記**する。実装者は自分のタスクしか見ないため、ここに書かない名前は存在しないのと同じ
- 全文提示→承認

## 失敗パターン

- **plan の丸写し**: コンポーネント一覧をそのままタスクにする。コンポーネントとタスクの単位は別（1コンポーネント複数タスクも、複数コンポーネント1タスクもある）
- **検証コマンドの手抜き**: 「テストを実行」とだけ書く。→ コマンドと期待出力まで書く。実行できないコマンドは analyze で差し戻される
- **依存の書き忘れ**: [P] を付けたタスク同士が実は同じファイルを触る。→ 変更対象の列で突き合わせ確認する
