# sdd-constitution — プロジェクト原則の策定

以後の plan / analyze / implement / review すべてが参照する「憲法」を作る。

## 手順

1. **既存情報を先に読む**: CLAUDE.md・README・リンター設定・既存コードの規約を確認し、矛盾しない案を作る。既存プロジェクトの慣習は原則より優先。
2. **1問ずつ確認**: 以下4節について、提案→ユーザー確認の順で1節ずつ確定する（選択式優先）。
   - 技術スタック方針（言語・フレームワーク・依存追加の基準）
   - 品質基準（テスト必須度、カバレッジ観、型チェック）
   - 禁止事項（例: 秘密情報のハードコード、テストなしのマージ）
   - レビュー方針（フェーズ承認は誰が・タスク完了の確認方法）
3. **書き出し**: このスキルと同じディレクトリの `template.md` をコピーして `docs/constitution.md` に置き、確定した内容で埋める（各項目の書き方・分量の規則は template のコメントに従う——二重管理しない）。
4. **承認**: 全文を提示して承認を得る。sdd 経由の場合、承認を state.md に記録するのは sdd の責務。

## 良い原則・悪い原則

| 悪い | なぜ | 良い |
|---|---|---|
| 高品質なコードを書く | 検算できない。analyze/review で使えない | 全受け入れ基準にテスト必須 |
| なるべく依存を減らす | 「なるべく」は判定不能 | 標準ライブラリで30行以内に書ける処理に依存を足さない |
| セキュリティに配慮する | 何をすれば準拠か不明 | 秘密情報のハードコード禁止・外部入力は全てバリデーション |

検算可能でない原則は plan の準拠チェック表を埋められない。書いた各項目に「準拠/逸脱を機械的に判定できるか？」を自問する。

## 原則

- 憲法は「守れる最小セット」にする。理想の羅列は形骸化する（YAGNI）
- 既に決まっていることを聞き直さない
- 途中で原則を変えたくなったら（実装中に窮屈だと判明した等）、黙って破らず constitution 自体の改訂をユーザーに提案する
