# implement — 実装フェーズ

## 前提チェック

tasks.md が承認済みでなければ sdd に差し戻す。
ただしバイパスルート（state.md に「ルート: バイパス」の記録あり）では tasks.md は不要。承認済み spec-lite.md を単一タスクとして扱い、本体セッションで直接実装する（1〜2ファイルのため）。
git 管理下で現在ブランチが main/master の場合、作業ブランチの作成をユーザーに確認してから着手する。承認済み成果物（spec/plan/tasks/state.md）がまだ未コミットなら、**作業ブランチ側で**最初にコミットする（main には直接コミットしない）。

## タスクごとの実行方式（自動選択）

- 変更対象が **1〜2ファイル** → 本体セッションで直接実装（TDD で）
- **3ファイル以上** → サブエージェント派遣
- 選択結果（サブエージェント | 直接）を state.md のフェーズ内進捗欄に記録する
- **直接実装のタスクも implementer-prompt の契約に従う**（TDD 証拠・セルフレビュー・報告フォーマット・タスク単位のコミット。BASE 記録も派遣時と同様に行う）。直接実装はタスクレビューを省略できるが、その分はフィーチャーレビュー（review フェーズ）で必ず拾われる——自己申告はレビューの代替にならない。**バイパス時は review フェーズがないため、verify 前に review のタスクレビューを1回通す**（sdd のバイパス節参照）

## サブエージェントのモデル選択

派遣時は必ずモデルを明示する（省略すると本体と同じ最上位モデルを継承し、コストが無駄になる）:
- タスク定義が完全（変更対象・完了条件・検証コマンドが具体的）な機械的実装 → **最安モデル**
- 複数ファイルの統合・既存コードの読解が必要 → **標準モデル**
- 設計判断を含む（本来 tasks が具体化すべきだった）→ 派遣せず、**tasks の粒度不足として差し戻す**

## サブエージェント派遣の規則

- **派遣前に現在のコミット（BASE）を記録**し、実装者にはタスク単位のコミットを義務付ける。タスクレビューの対象 diff は BASE..HEAD で取る（working tree 比較では前タスクの変更が混ざる。`HEAD~1` は複数コミットのタスクを切り落とすので使わない）
- 渡すもの: `implementer-prompt.md` の雛形＋該当タスク定義（tasks.md の当該節のみ）＋spec.md の関連 AC ＋constitution.md
- **渡さないもの**: plan 全文・他タスク・会話履歴（コンテキスト隔離。タスク定義で完結させる）。過去タスクの経緯の貼り付けは膨張の元——新しい実装者に必要なのは自分のタスクと隣接インターフェースだけ
- `[P]` タスクは同時派遣可。ただし同一ファイルを触るタスクの同時派遣は禁止
- タスク完了報告を受けたら、**本体が検証コマンドを自分で実行**して確認し、state.md のチェックボックスを更新する。サブエージェントの「できました」を鵜呑みにしない
- サブエージェントに派遣したタスクは、検証に加えて review（タスクレビューモード。対象は当該タスクの diff）を通し、重大・重要の指摘を修正してから次のタスクへ進む
- レビュー指摘の修正は**同じ実装者に戻す**（文脈を持っているため）。本体が自分で直すのはコンテキスト汚染になるので避ける。修正後は必ず再レビュー

## 実装者のステータス契約と対応

実装者は4種のステータスで報告する。それぞれの対応:

| ステータス | 意味 | 本体の対応 |
|---|---|---|
| DONE | 完了・検証済み | 本体検証 → タスクレビュー → 進捗更新 |
| DONE_WITH_CONCERNS | 完了したが懸念あり | 懸念を読む。正しさに関わる懸念は解消してからレビューへ。観察系（「ファイルが大きくなってきた」等）は記録して進む |
| NEEDS_CONTEXT | 情報不足 | 不足情報を渡して再派遣（同じモデル） |
| BLOCKED | 完了できない | 下のエスカレーション梯子へ |

**エスカレーション梯子**（BLOCKED・失敗時。この順に1段ずつ）:
まず BLOCKED の理由を読む。**理由が仕様矛盾なら梯子に乗せず「仕様矛盾を発見したら」節へ直行**。それ以外は
①文脈不足なら情報を足して再派遣 → ②推論力不足なら1段上のモデルで再派遣 → ③タスクが大きすぎるなら分割 → ④タスク定義自体が誤りなら tasks へ差し戻し（ユーザー報告）。
同じ条件・同じモデルでの単純リトライは禁止。実装者が「詰まった」と言ったら何かを変える。
同じタスクで2回失敗したら本体で引き取り、原因を特定してからタスク定義を修正する。
**直接実装でも同じ契約を自分に適用する**: BLOCKED 相当（設計判断の分岐・仕様矛盾・確信のないまま書き進めている）に気づいたら、作業を止めて state.md に記録しユーザーに報告する。「自分が書いているから続行してよい」は禁止。

## 仕様矛盾を発見したら

実装を止めて報告する。直す前に**どちらが正かを判定する**: ユーザーの元依頼と承認済みの受け入れ基準に照らし、要求を正しく表している方を残す（spec が誤っていることも plan が誤っていることもある。「矛盾＝spec の欠陥」と思い込まない。判定に迷ったらユーザーに聞く）。
誤っていた側（spec.md または plan.md）を修正（ユーザー承認必要）→ 差し戻し履歴に記録 → 下流成果物（tasks 等）の更新要否を確認 → **修正が AC やタスクの対応関係に及んだ場合は analyze を再実行**してから再開。黙って仕様と違うものを作らない。

## 進行の規律

- タスク間で「続けますか？」と確認しない。止まるのは BLOCKED を解消できないとき・仕様矛盾・全タスク完了のときだけ
- タスクごとの進捗は state.md のチェックボックスが正。会話の記憶に頼らない（コンテキスト要約後は state.md を信じる）

## フェーズ完了

全タスクのチェックボックスが埋まったら、実行方式の内訳と検証結果の一覧を提示して承認を得る。
