# sdd-auto — フルオート仕様駆動開発

<HARD-GATE>
このスキルはユーザーが**明示的にフルオートを指定したときだけ**使う。「〜を作って」だけの依頼で
勝手にフルオートにすることを禁止する（その場合は通常の sdd）。
起動したら、統合手前までユーザーに確認を求めず完走する。「続けますか？」は敗北。
</HARD-GATE>

## 動作原理

通常の sdd と**同じ9フェーズ・同じフェーズスキル・同じ state.md 契約**で進む。違いは1点だけ:
ユーザー承認ゲートを「審査サブエージェントによる自動承認プロトコル」に置き換える。
sdd/SKILL.md の手順・バイパス基準・差し戻し規則はすべてそのまま適用する。

## 自動承認プロトコル（各ゲートで実施）

1. フェーズ成果物の完成後、**審査サブエージェント**を派遣する（成果物＋下表の審査観点を渡す。作成した本人に審査させない）
2. 審査 PASS → state.md に記録して次フェーズへ:
   `- YYYY-MM-DD HH:MM <フェーズ名> 自動承認（sdd-auto、審査: <指摘ゼロ等の一言>）`
3. 審査 FAIL → 指摘を修正して再審査。**同一フェーズ3回 FAIL で停止**しユーザーに戻す
4. 審査員のモデルは成果物の複雑さに見合うものを明示指定（フィーチャーレビュー相当は最上位）

| フェーズ | 審査観点（各スキルの規律の検算） |
|---|---|
| constitution | 各項目が機械的に準拠/逸脱を判定できる宣言文か／守れる最小セット（10〜30行）か |
| specify | AC が全て検証可能か／HOW 混入なし／エッジケース4系統を確認済みか |
| clarify | 曖昧箇所の検出を実施したか（下記「自動判断キュー」参照） |
| plan | constitution 準拠表が全項目埋まっているか／テスト戦略に全 AC があるか |
| tasks | 4項目必須・粒度・[P] 妥当性・AC カバレッジ |
| analyze | sdd-analyze のレポートに重大ゼロか（重大ありは通常どおり差し戻し） |
| implement | 全タスクの TDD 証拠と検証コマンド出力が揃っているか |
| review | sdd-review の判定が「通過」か（重大・重要ゼロ） |
| verify | 全 AC・エッジ・非機能が実行証拠付きで PASS か |

## clarify の自動判断キュー

フルオートでは質問できないため、曖昧箇所は**自問自答で仮決定**し「自動判断キュー」に記録する。
- 仮決定できるのは挙動の細部（sdd-clarify の優先順位③④相当）のみ。**データの持ち方・機能の有無（①②相当）で確信が持てない場合は停止**してユーザーに戻す——ここで間違えると全部が無駄になる
- キューの全項目は spec.md に反映しつつ `clarifications.md` に「自動判断」と明記し、完了報告で一覧提示する（ユーザーが後から覆せるように）

## 収束ループ（ループエンジニアリングの核）

implement → review → verify は一発で通ることを期待しない。FAIL・差し戻しを前提に回す:

```
implement 完了 → review（指摘→修正→再レビュー） → verify（FAIL→implement差し戻し→再review→再verify）
```

- 修正ループの回数を state.md のフェーズ内進捗欄にカウントする
- **ワークフロー全体で修正ループ合計10回**を超えたら停止（収束していない＝仕様か計画に構造的問題がある。ループで殴り続けない）
- 各ループの修正は指摘への対処だけを行う。「ついでの改善」はループを発散させる

## 必ず停止してユーザーに戻す条件（フルオートでも越えない一線）

1. 破壊的操作が必要になった（既存資産の削除・上書き・force push・外部サービスへの送信）
2. 仕様の根本的曖昧さ（自動判断キューの①②相当）や、実装中に発覚した仕様矛盾で spec の書き直しが必要
3. constitution からの逸脱なしに実装できないと判明した
4. 同一フェーズ審査3回 FAIL、または修正ループ合計10回到達
5. **統合（main へのマージ・PR 作成・push）**——検証が全件 PASS でも、ここから先は必ずユーザーが選ぶ

停止時は「どこまで進んだか（state.md）・何に詰まったか・選択肢」の3点で報告する。

## 完了報告（統合手前で必ず提示）

- 検証レポート（N/N PASS、証拠付き）
- 自動承認ログ（state.md 承認履歴の写し）
- 自動判断キューの一覧（覆すなら差し戻せる旨を添える）
- 修正ループの回数と内訳
- 統合3択（sdd-verify 手順7と同じ。ユーザーが選ぶ）
