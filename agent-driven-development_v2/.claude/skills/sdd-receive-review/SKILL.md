# sdd-receive-review — 受領レビューへの対応

PR コメント・ユーザー指摘など**外から受け取ったレビュー**を処理する。
sdd-review（自分がレビューする側）とは逆方向のスキル。

<HARD-GATE>
指摘を検証せずに実装することを禁止する。「ご指摘ありがとうございます、直します」の脊髄反射は、
正しい仕様を壊す・不要な変更を持ち込む事故の元。同意は検証の後。
</HARD-GATE>

## 手順1: 指摘の検証（実装より先）

各指摘を1件ずつ、コードと spec.md に照らして検証し、4分類する:

| 分類 | 判定基準 | 対応 |
|---|---|---|
| 妥当（実装品質） | 指摘どおりの欠陥がコードにあり、AC には影響しない | 修正ルートへ（手順3） |
| 妥当（仕様変更） | 指摘は正しいが、直すと既存 AC の期待値・挙動が変わる | AC 修正の承認を先に取ってから修正ルートへ |
| 不当・誤解 | 検証した結果、指摘が技術的に誤り／仕様どおりの挙動 | 反証（コード・spec の根拠付き）を用意し、ユーザーに提示して見送りの承認を得る。黙って無視しない |
| 不明瞭 | 何を求めているか2通り以上に読める | 推測で実装せず、質問を返す |

検証せずに全指摘を「妥当」に分類したくなったら、それは追従バイアス。1件ごとに「この指摘の根拠は何か・反例はないか」を自問する。

## 手順2: 機能の再オープン

- 指摘が紐づく `specs/NNN-機能名/` を特定する（複数機能にまたがる場合は機能ごとに分ける）
- state.md の現在フェーズを「完了」から戻し先フェーズに書き換え、差し戻し履歴に記録する:
  `- YYYY-MM-DD HH:MM 再オープン: 完了→<フェーズ>（理由: PR レビュー指摘 <要約>）`
- 同一機能への複数指摘は、まとめて1回の再オープンで扱う（指摘ごとに開閉しない）

## 手順3: 修正ルートの振り分け（指摘ごとに判定）

- **実装品質・1〜2ファイル** → sdd のバイパスルート（spec-lite → implement → verify 前タスクレビュー → verify）
- **仕様変更を含む** → sdd の救済ルート（該当 AC を承認付きで修正→バイパス続行）。変更が大きければフルルートに差し戻し
- **設計レベル**（コンポーネント分割・データモデルに及ぶ） → plan へ差し戻し（sdd の差し戻し手順どおり）

## 手順4: 完了と返信

- 修正後は verify を再実行する（修正が触れた AC は必ず、他は影響範囲で判断）。証拠付きで全件 PASS を確認してから完了
- PR には指摘ごとの対応結果を返す: 修正した（コミット参照）／反証して見送り（理由）／質問。**対応しなかった指摘を無言で放置しない**
- verify の完了処理で state.md が「完了」に戻っていることを確認する（「完了」への更新は sdd-verify の責務）

## 失敗パターン

- **追従実装**: 検証を飛ばして指摘どおりに直す → 正しい仕様が壊れる（HARD-GATE 違反）
- **黙殺**: 不当と判断した指摘に何も返さない → レビュアーとの信頼が壊れる。必ず反証か質問を返す
- **spec の置き去り**: コードだけ直して spec.md の AC が古いまま → 次の開発で矛盾として爆発する。仕様変更を含む修正は AC を先に直す
