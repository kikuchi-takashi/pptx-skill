# analyze — 整合性検証（read-only）

<HARD-GATE>
このスキルは成果物を書き換えない。報告のみ。修正は該当フェーズへの差し戻しで行う。
</HARD-GATE>

## 検証項目（すべて実施）

1. **カバレッジ表**: 全 AC-n × タスクの対応表を作る。どのタスクにも現れない AC は「漏れ」
2. **spec↔plan 矛盾**: plan のコンポーネント・データモデルが spec の受け入れ基準と食い違っていないか
3. **constitution 違反**: plan・tasks が constitution の各項目に反していないか（plan の準拠チェック節の検算）
4. **検証コマンドの実行可能性**: tasks.md の各検証コマンドが現環境で実行できる形式か。ツールの存在は実際に確認する（`which` やバージョン表示を実行する。推測しない）。存在しないツールは重大（実行不能）として報告する
5. **タスク間インターフェース**: あるタスクが使う関数・型を、依存先タスクの完了条件が定義しているか（名前・シグネチャの不一致は実装時に必ず衝突する）

## 検出例（各カテゴリ1つ）

- 漏れ: AC-4「空入力で再入力を促す」がどのタスクの完了条件にもない
- 矛盾: spec は「保存は即時反映」、plan のデータモデルは「終了時に一括書き出し」
- constitution 違反: 「依存追加は最小限」に対し、plan が標準ライブラリで書ける処理に外部ライブラリを選定
- 実行不能: 検証コマンドが `npm test` だがプロジェクトに package.json がない
- インターフェース不一致: T3 が `save(task)` を呼ぶ前提だが、T1 の完了条件は `save(task_list)` を定義

## 重大度の較正

- **重大** = 実装に入ると手戻りが確実なもの（AC 漏れ・spec↔plan 矛盾・インターフェース不一致・実行不能な検証コマンド）
- **軽微** = 実装中に吸収できるもの（表記ゆれ・冗長な記述・粒度の不揃い）
- 迷ったら「これを放置して実装した場合、どのフェーズまで戻るか」で判定する。tasks 内で直る＝軽微、spec まで戻る＝重大
- 判定の第一基準は**テスト可能な挙動への影響**。文言の精度だけの問題（挙動は変わらない）は軽微とし、指摘には残す

## 報告フォーマット

```markdown
## 整合性レポート
### カバレッジ表
| AC | タスク | 状態 |
### 問題（重大度順）
- [重大|軽微] <内容> → 差し戻し先: <フェーズ>
### 判定: 実装に進んでよい | 差し戻しを推奨
```

重大問題ゼロでも必ずレポート全文を提示し、ユーザーの確認を得てから implement へ進める（承認記録は sdd の責務）。
「問題なし」の1行報告は禁止——カバレッジ表と確認した項目を示すこと。何を見てゼロと言ったかが分からない報告は検証ではない。
