# sdd-specify — 仕様書の作成

## HOW 禁止ルール

仕様書に次を書いたら失敗: フレームワーク名・ライブラリ名・DB 種別・API 設計・ファイル構成・クラス名。
「ユーザーがログインできる」は仕様。「JWT でセッション管理する」は仕様ではない（plan の仕事）。
書き終えたら技術名詞（言語名・ライブラリ名・ファイル拡張子・コマンド名）を検索して自己チェックする。

## 出力フォーマット（specs/NNN-機能名/spec.md）

このスキルと同じディレクトリの `template.md` をコピーして埋める。テンプレート内のコメントが記入ガイド。
項目の追加は自由。項目を削除する場合は「なし」と書くか削除理由を1行残す。

## 受け入れ基準の良い例・悪い例

| 悪い（検証不能） | 良い（検証可能） | なぜ |
|---|---|---|
| 使いやすい UI にする | 主要3操作が各3クリック以内で完了する | 「使いやすい」は人によって違う |
| エラーを適切に処理する | 不正な入力では終了せず、原因を1行で表示して再入力を促す | 「適切」は判定できない |
| 高速に動作する | 1000件の入力で1秒以内に結果を返す | 数値がなければ合否が出せない |
| データを安全に保存する | 保存中に強制終了しても既存データは壊れない | 「安全」の中身を挙動で言う |

Then が「〜される」で終わるときは「誰がどう観測できるか」を足す。観測手段のない Then は検証できない。

## エッジケースの探し方

各ユーザーストーリーに対して最低限この4系統を確認し、該当するものをエッジケース節に書く:
①空・ゼロ（入力なし・0件・空文字） ②境界（最大値・最小値・ちょうど上限） ③重複・同時（二重送信・同名・並行操作） ④異常系（不正な型・存在しない参照・権限なし）。

## 手順

1. 依頼内容からユーザーストーリーを起こす（1機能につき1〜5個、P1〜P3 の優先度付き）。6個以上になるなら機能が大きすぎる——sdd に分解を提案する
2. 各ストーリーに受け入れ基準を AC-番号付きで書く。**検証可能な表現のみ**（上の表を基準にする）
3. 判断できない点は本文に `[要確認: 〜]` マーカーを埋め込む。**推測で埋めるのが最悪の失敗**。もっともらしいデフォルトを書いて黙って進むより、マーカーを残して clarify に回す
4. スコープ外を明記する（YAGNI の防波堤）。「頼まれていないが良さそうな機能」を思いついたら、実装に足すのではなくスコープ外に書く
5. 全文を提示して承認を得る。`[要確認]` が残っていれば「次は clarify」と伝える
