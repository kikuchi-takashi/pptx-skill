# sdd-maintain — スキルセットのメンテナンス

SDD スキルセットを変更するときは必ずこのスキルを通す。個々のスキルは独立したファイルだが、
**互いに契約で結ばれた1つのシステム**である。1ファイルだけ見て直すと必ず境界が壊れる
（実績: 過去の全監査で、指摘は単一スキルの欠陥ではなく「スキル間の境界」に集中した）。

## 壊してはいけない契約（変更前に必ず確認）

| 契約 | 正（single source of truth） | 変更時に追随が必要な場所 |
|---|---|---|
| state.md 書式（列挙値・承認履歴3形式・差し戻し履歴3形式・フェーズ内進捗） | sdd/SKILL.md | 全フェーズスキルの state.md 言及箇所・sdd-auto |
| フェーズ順序と数（現在9: constitution→…→verify） | sdd のフェーズ表 | sdd の description「9フェーズ」・列挙値・CLAUDE.md フェーズ図・sdd-auto 審査観点表 |
| description 規約 | この表の下の段落 | 新規スキル全部 |
| テンプレート同梱 | 各スキル配下の template.md | SKILL.md は参照のみ（規則の二重管理禁止） |
| 承認の記録責務 | sdd（各フェーズスキルは記録しない） | 新フェーズスキルにも「記録は sdd の責務」を明記 |
| バイパスルート（テスト込み1〜2ファイル・新規採番・verify 前タスクレビュー） | sdd のバイパス節 | sdd-implement・sdd-verify・sdd-auto・spec-lite-template |

**description 規約**: frontmatter は `name:`（kebab-case、ディレクトリ名と一致）と `description:`（二重引用符付き1行）の2キーのみ。description には「何をするか＋日本語トリガー例＋（フェーズスキルなら）『通常は sdd オーケストレーター経由で呼ばれる。単体でも呼び出し可』」。本文は日本語・500行以内。

## 変更の種類別チェックリスト

**A. 既存スキルの修正**（最頻・低リスク）
1. 修正が上の契約表のどれかに触れるか確認。触れるなら「追随が必要な場所」を同一コミットで更新
2. 判断を変えない飾り文・既出の再掲を足さない（増強するときも「この記述がないと現場で判断を誤るか？」で選別）

**B. フェーズの追加・削除**（最重リスク。review フェーズ追加時の実績手順）
1. sdd: フェーズ表に行を追加/削除・description のフェーズ数と列挙・state.md 列挙値
2. 新フェーズの SKILL.md（description 規約・前提チェック・差し戻し規則を含める）
3. sdd-auto: 審査観点表に行を追加
4. CLAUDE.md: フェーズ図と構成
5. 隣接フェーズ: 前後のスキルに新フェーズへの受け渡しを明記

**C. 独立スキルの追加**（sdd-maintain 自身のような非フェーズスキル）
- description に「フェーズスキルではないため sdd からは呼ばれない」相当の一文を入れる（sdd の自動発火と競合させない）
- CLAUDE.md の構成に1行追加

**D. テンプレートの変更**
- テンプレートに項目を足したら、それを検証する側（sdd-analyze の検証項目・sdd-auto の審査観点・sdd-verify の手順）に検算が必要か確認

## 変更後の必須検証（コミット前）

```bash
cd <スキルセットのルート> && for d in .claude/skills/*/; do n=$(basename "$d"); \
  { head -1 "$d/SKILL.md" | grep -qx -- '---' && grep -q "^name: $n$" "$d/SKILL.md" \
    && grep -q '^description: "' "$d/SKILL.md"; } || echo "NG: $n"; done; echo FM-DONE
```

加えて: ①旧記述の残存を grep（フェーズ数表記・改名した節名・削除した規則への参照） ②description を変えた場合は発火シミュレーション（サブエージェントに「ユーザーが『〜』と言ったらどのスキルか」を判定させる）。

## 改善ループの回し方（実績のある3型）

どの型も「指摘→反映→コミット」を1周として回す。修正は指摘への対処だけを行い、ついでの改善で発散させない。

**① 批評レビュー**: 最上位モデルのサブエージェントに、参考実装（superpowers 等）との直接比較で「欠け・矛盾・水膨れ」を最大10〜12件、修正案付きで挙げさせる。read-only。
**② 実戦トライアル**: スキル一式をサンドボックス（一時ディレクトリ＋git init）へコピーし、操縦者サブエージェントに小さな題材で実走させる。**摩擦ログが最重要成果物**——【迷った/矛盾/不足/過剰/効いた】の5分類＋スキル名:節名で記録させる。正常系・フルオート・異常系（再開/差し戻し/state 破損/バイパス切替）の題材を使い分ける
**③ 整合性監査**: パッチが積もったら（目安: 20箇所超の変更後）、最上位モデルに全ファイル横断で「継ぎ接ぎ矛盾・古い記述の残存・契約違反」を監査させる。上の契約表を監査観点として渡す

**収束判定**: 指摘が「新しい構造的欠陥」から「境界の未定義の穴埋め」に変わったら収穫逓減——ループを止め、実利用からのフィードバック待ちに切り替える。

## 失敗パターン

- **1ファイル修正**: 契約表を見ずに単一スキルだけ直す → 監査で境界矛盾として検出される（最頻の欠陥源）
- **自己審査**: 自分が書いた変更を自分だけでレビューして終える → 批評・監査は必ず別のサブエージェントに出す
- **ループの空転**: トライアルの摩擦ログなしに「たぶんこう直すと良い」で変更する → 変更は必ず指摘（批評・摩擦・監査）に紐付ける
