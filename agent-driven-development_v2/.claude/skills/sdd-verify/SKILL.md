# sdd-verify — 受け入れ検証と完了

<HARD-GATE>
証拠（実際に実行したコマンドとその出力）なしに「完了」「動きます」と報告することを禁止する。
</HARD-GATE>

## 禁止フレーズ

これらを書きそうになったら、その主張の証拠をまだ持っていない:

| 書きそうな言葉 | 実際にやること |
|---|---|
| 「実装したので完了です」 | 実装と完了は別。AC を実行して初めて完了 |
| 「テストが通ったので大丈夫」 | ユニットテストのパス ≠ 受け入れ基準の充足。AC の Then を直接実証する |
| 「たぶん動くはず」「〜のはず」 | 推測を報告に書かない。実行して確かめる |
| 「概ね問題ありません」 | 概ね＝FAIL がある。N/M PASS と正直に書く |
| 「前回動いたので今回も動く」 | 変更後の状態で再実行する |

## 手順

1. spec.md（バイパス時は spec-lite.md）の受け入れ基準を1件ずつ、**Then を実証するコマンドや操作を実際に実行**して確認する。ユニットテストの再実行で代用しない——AC はユーザーから見える挙動で書かれているので、その挙動を直接動かす
   検証で作る一時ファイル・一時データはプロジェクト外（OS の一時ディレクトリ等）に置くか、`.sdd-verify-tmp/` にまとめて終了時に削除する。プロジェクト直下に散らすと `git add -A` で混入する
2. 結果を検証レポートとして記録する:

```markdown
## 検証レポート
### AC-1: <名前> — PASS | FAIL
- 実行: `<コマンド>`
- 出力: <そのまま貼る>
```

3. FAIL があれば implement へ差し戻す（差し戻し履歴に記録）。差し戻し修正の diff は sdd-review（タスクレビュー）を通してから再検証する——修正が新しい問題を持ち込むため review を素通りさせない。全件 PASS まで完了と言わない。**部分成功は「N/M PASS、FAIL は AC-x」と正直に報告する**——FAIL を伏せた完了報告は虚偽報告
4. spec のエッジケース節・非機能要件節も同様に1件ずつ実証する（「なし」と明記されている節はスキップ可）
5. sdd-review から持ち越された軽微指摘の一覧があれば、完了報告に添えて採否をユーザーに委ねる
6. 全件 PASS: state.md の現在フェーズを「完了」にし、レポート全文を提示する
7. 統合方法の選択肢を提示する（ユーザーが選ぶ。勝手に実行しない）:
   - ① **ローカルマージ**: 作業ブランチを main へマージし、マージ後にテスト一式を再実行して確認
   - ② **PR 作成**: 作業ブランチを push して Pull Request を作る（ブランチは残す）
   - ③ **そのまま**: 作業ブランチにコミット済みの状態で終了（統合はユーザーが後で行う）
