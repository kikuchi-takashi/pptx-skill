# sdd — 仕様駆動開発オーケストレーター

開発依頼を受けたら、コードを1行も書く前にこのワークフローで仕様から始める。

<HARD-GATE>
承認記録のないフェーズを飛ばして先のフェーズスキルを呼ぶことを禁止する。
フェーズ成果物を作ったら必ず要約を提示し、ユーザー承認を得てから state.md に記録して次へ進む。
</HARD-GATE>

## 合理化の罠

次の考えが浮かんだら、それはワークフローを飛ばす言い訳である。止まれ。

| 浮かんだ考え | 現実 |
|---|---|
| 「簡単だから仕様は不要」 | 簡単な案件ほど未検証の思い込みで手戻りする。仕様は数行でよいから書く |
| 「要件は明らかだから clarify 不要」 | 明らかに見える要件が2通りに読めることは頻繁にある。マーカーゼロの確認だけでもする |
| 「まずコードを書いて見せた方が早い」 | 動くものを見せると仕様の議論ができなくなる。プロトタイプが必要なら spec にその旨を書く |
| 「前回と同じパターンだから analyze 省略」 | 整合性検証は「同じに見えて違う」を検出するためにある |
| 「ユーザーは急いでいるから承認を待たずに進む」 | ゲートを飛ばした分の手戻りの方が遅い。承認は非同期に取ればよい |

## 起動時の手順（必ずこの順）

1. **再開チェック**: `specs/*/state.md` を走査する。「現在フェーズ」が「完了」以外の機能があれば一覧を提示し、再開するか新規かをユーザーに確認する。再開なら該当フェーズのスキルを呼ぶ。再開時はそのフェーズの成果物を開き、「手つかず／書きかけ（テンプレートの空欄・[要確認] の残存）／完成済み・未承認」のどれかを確認してから続きに入る。**現在フェーズが implement の場合はこの3分類をそのまま使わない**——成果物がテンプレート1枚ではなく tasks.md のチェックボックス＋実コード＋テストの集合体であるため。tasks.md のチェックボックスをそのまま信じず、各タスクについて実コードの存在と検証コマンドの再実行結果で完了/未着手を裏取りしてから続きに入る。
2. **constitution チェック**: `docs/constitution.md` がなければ、先に constitution スキルを呼ぶ。constitution の承認はプロジェクトで一度だけ（作成した機能の state.md に記録。state.md は手順5のフォルダ作成で作られるため、記録はフォルダ作成後に1件目の承認履歴として遡って書く）。2機能目以降は既存の constitution をそのまま使い、再承認は不要。
3. **スコープ判定**: 依頼が複数の独立したサブシステムを含む場合（例:「チャットとファイル共有と課金があるアプリ」）、1つの spec に詰め込まない。独立した部分・相互関係・構築順を提示して分解し、最初のサブシステムから通常フローに入る。サブシステムごとに spec→plan→実装のサイクルを回す。
4. **ルート判定**: 下記のバイパス基準で フル / バイパス を判定し、ユーザーに判定結果と理由を伝える。
5. **作業ブランチとフォルダの作成**: 現在ブランチが main/master なら、**先に作業ブランチ（例: `feature/YYYY-MM-DD-機能名`）を作る**——この機能の specs/ も実装コードも最初から**すべて同じブランチ上で生む**（specs/ を main や worktree 外に取り残さない）。worktree で実装を回す運用（implement のサブエージェントを worktree 隔離で派遣する等）では、**この時点で worktree を作り、以降の全フェーズをその worktree 内で実行する**——specify〜analyze の成果物を元の作業ディレクトリに置くと、worktree 側のブランチ＝PR に含まれない。続けて `specs/YYYY-MM-DD-機能名/`（日付は開始当日、機能名は kebab-case）を作成し state.md を初期化する。同名フォルダが既にある場合は機能名を具体化して区別する（例: `-2` を付けるのではなく `todo-cli-export` のように内容で分ける）。フルオートでは作業ブランチ作成の確認を省き無確認で作る（auto 参照）。
6. **フェーズ実行**: 現在フェーズに対応するスキルを Skill ツールで呼ぶ。フェーズ完了→承認→state.md 更新→次フェーズ、を繰り返す。

## フェーズ順序と対応スキル

| # | フェーズ | スキル | 承認 |
|---|---------|--------|------|
| 1 | constitution | constitution | 必須（プロジェクト初回のみ） |
| 2 | specify | specify | 必須 |
| 3 | clarify | clarify | 曖昧箇所ゼロなら自動通過可（通過も state.md に記録） |
| 4 | plan | plan | 必須 |
| 5 | tasks | tasks | 必須 |
| 6 | analyze | analyze | 必須（重大問題ゼロの報告に対する確認） |
| 7 | implement | implement | タスクごとに検証、フェーズ完了時に承認 |
| 8 | review | review | 必須（重大・重要の指摘ゼロで通過） |
| 9 | verify | verify | 必須（証拠付き完了報告） |

## ゲート運用

- 承認を求めるときは「成果物の要約（5〜10行）＋全文の場所＋確認してほしい論点」をセットで提示する。全文を貼り直すだけの丸投げをしない
- 承認は「はい/OK/承認」等の明示的な返答のみ。無言・話題転換は承認ではない
- ゲートの自動化は、ユーザーが明示的にフルオートを指定して **auto** 経由で起動した場合のみ許される。sdd が自分の判断でゲートを自動承認に切り替えることは禁止
- フェーズ内の作業は途中で「続けますか？」と聞かずに完遂する。止まるのはゲート・ブロック・ユーザーにしか決められない選択肢が出たときだけ
- **承認済みの成果物（spec/plan/tasks/analyze-report/state.md 等）は、承認のたびに作業ブランチへコミットする**。implement まで未コミットで溜めない——worktree で実装を回す場合、未コミットの specs/ は worktree に伝わらず PR に含まれない。単一ディレクトリ運用でも随時コミットしておくと復旧材料になる

## 小規模バイパス基準（すべて満たす場合のみ）

対象は**既存 spec.md を持つ機能への追加・修正**。spec.md 自体がまだ無い新規機能（既存の仕様書が存在しない依頼）は、条件2「既存spec.mdの受け入れ基準に影響しない」を判定する土台がないためバイパス対象外——必ずフルルートで specify から開始する。

- 変更が1〜2ファイルに収まる見込み（**テストファイルも数に含める**。既存テストの期待値更新もこの変更の一部）
- 既存 spec.md の受け入れ基準に影響しない（**AC の期待値を変える依頼は「影響する」に該当**——デフォルト値の変更等も含む）
- 新しい依存関係・データモデル変更を伴わない

条件2だけに抵触し、変更自体は1〜2ファイルの場合の救済ルート: **先に既存 spec の該当 AC を修正してユーザー承認を取り**（該当機能の state.md 差し戻し履歴に「仕様修正: spec」で記録）、その上でバイパスを続行してよい。承認なしに古い spec と矛盾する実装を作らない。

該当する場合: specify/plan/tasks を `spec-lite.md`（①変更内容 ②受け入れ基準 ③検証コマンド の3節。雛形: このスキルと同じディレクトリの `spec-lite-template.md`）1枚にまとめ、ユーザー承認後 implement → verify へ直行する（review フェーズ自体は省略するが、**verify の前に実装 diff を review のタスクレビューで1回通す**——バイパスでは他にコードレビューの機会がないため）。バイパス採用と理由を state.md に記録する。
バイパスでも `specs/YYYY-MM-DD-機能名/` は**新規に作成**する（既存機能のフォルダに追記しない。既存 spec への影響がないことがバイパスの条件なので、独立した記録として残す）。例外は receive-review 経由の再オープン——この場合は新しいフォルダを作らず、再オープンした既存フォルダ内に spec-lite を追加する。
バイパス中の「現在フェーズ」は、spec-lite 作成中は specify、実装中は implement、検証中は verify を用いる。**「verify の前に実装 diff を review のタスクレビューで1回通す」工程の最中は review を用いる**（review フェーズ自体は省略するが、現在フェーズの値としては列挙値の review をそのまま流用する）。
**実装中に変更が3ファイル以上に膨らんだら、バイパスは誤判定だった**。実装を止めてフルルートに切り替える:
1. 書いた分の扱い: **テストは spec の入力として残す**（発見した挙動・エッジケースを AC の材料にする）。実装コードは原則捨てる（フルルートの TDD で書き直す方が安全）
2. spec-lite → spec の昇格: 「変更内容」→ 背景・目的とユーザーストーリーに展開／「受け入れ基準」→ AC-n に番号付けして移設／「検証コマンド」→ plan のテスト戦略に持ち越す
3. state.md: 「ルート」を フル に書き換え、差し戻し履歴に「ルート変更: バイパス→フル（理由: 〜）」を記録する。specs/ の既存フォルダはそのまま使う（新しいフォルダを作らない）

## state.md フォーマット（全フェーズスキル共通の正）

```markdown
# YYYY-MM-DD-機能名 進行状態
- 機能名: <日本語名>
- ルート: フル | バイパス（理由: 〜）
- 現在フェーズ: constitution | specify | clarify | plan | tasks | analyze | implement | review | verify | 完了
## 承認履歴
- YYYY-MM-DD HH:MM <フェーズ名> 承認（ユーザー）
- YYYY-MM-DD HH:MM clarify 自動通過（曖昧箇所ゼロ）
- YYYY-MM-DD HH:MM <フェーズ名> 自動承認（auto、審査: <一言>）
## フェーズ内進捗（implement 中とフルオートの修正ループ中のみ）
- [ ] T1: <タスク名>（サブエージェント | 直接）
## 差し戻し履歴
- YYYY-MM-DD HH:MM <戻したフェーズ> 理由: 〜
- YYYY-MM-DD HH:MM ルート変更: バイパス→フル（理由: 〜）
- YYYY-MM-DD HH:MM 仕様修正: spec|plan を修正（理由: 〜）
- YYYY-MM-DD HH:MM 再オープン: 完了→<フェーズ>（理由: 〜）
```

上記は完成形の一枚絵であり、実際の更新は**差分追記**で行う。承認履歴・差し戻し履歴は1行ずつ末尾に追記する。「## フェーズ内進捗」見出しは implement 開始時（またはフルオートの修正ループ開始時）に**初出のときだけ**追加し、既に存在する見出しの下にチェックボックスを追記する（見出しを重複させない）。
再オープンしたサイクルだけが元のルートと異なる進め方（例: 当初フルルートの機能をPR対応だけバイパスで進める）をする場合、「ルート」欄は当初の値のまま据え置き、差し戻し履歴の「再オープン」行に今回サイクルの進め方を明記する（例: 「…理由: 〜／今回サイクルはバイパスで対応」）。ルート欄自体を書き換えない。

「現在フェーズ」には上記の列挙値のみを書く（注釈・補足を付けない。補足は承認履歴か差し戻し履歴の行に書く）。
state.md は会話の記憶より優先する。コンテキストが要約・切断された後は、自分の記憶ではなく state.md と成果物ファイルを信じる。

## 差し戻し

どのフェーズからでも前フェーズへ戻せる。戻す際は理由を差し戻し履歴に記録し、下流成果物（例: spec 変更時の plan/tasks）の更新要否をユーザーに明示する。
**完了した機能も再オープンで戻せる**（PR レビュー指摘などの受領レビュー対応は receive-review 経由）。再オープン時は現在フェーズを「完了」から戻し先に書き換え、差し戻し履歴に「再オープン」形式で記録する——再開チェックは「完了以外」を走査するため、書き換えないと再開対象に載らない。

## state.md が壊れている・矛盾している場合

成果物ファイル（spec.md / plan.md / tasks.md / **analyze-report.md / review-report.md**）に加えて、**実コード・テストファイル・git log** を推定材料にする。analyze-report.md の判定・review-report.md の判定（通過 | implement へ差し戻し）は「そのフェーズが本当に完了したか、途中で止まっているか」を区別する決め手になることが多い——spec/plan/tasksだけでは analyze・review を通過済みかどうか判断できない。テストが存在してパスするタスクは完了と推定できる——ファイルの「存在」より強い証拠を優先する。
ファイルの存在は承認の証明にならない。承認済みかは「[要確認] マーカーが残っていないか・テンプレートの空欄がないか・差し戻しコメントが残っていないか」で判定する。
承認履歴の**時刻**は git log のコミット時刻で代用できるとは限らない（作業順序の証跡であって、元の承認時刻と一致する保証はない）。時刻が推定できない行は日時を空欄にせず `[復旧・時刻不明]` を付記し、正確な時刻ではなく推定であることを明示する。
推定結果をユーザーに確認してから state.md を書き直す。黙って上書きしない。
