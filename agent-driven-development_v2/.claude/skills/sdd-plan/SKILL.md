# sdd-plan — 技術計画

## 前提チェック

spec.md が承認済み（state.md の承認履歴に記録あり）でなければ、sdd に差し戻す。`[要確認]` が残っていたら clarify に差し戻す。

## 出力フォーマット（specs/NNN-機能名/plan.md）

このスキルと同じディレクトリの `template.md` をコピーして埋める。テンプレート内のコメントが記入ガイド。
テスト戦略の表にはすべての AC-n を、constitution 準拠チェックの表には constitution の全項目を列挙する。

## 大きな選定は複数案で提示する

アーキテクチャを左右する選定（データの持ち方・主要ライブラリ・構成の骨格）は、**2〜3案をトレードオフ付きで挙げ、推奨案と理由を先頭に置いて**ユーザーに提示してから plan に書く。
些末な選定（変数命名・軽微なユーティリティ）は1案でよい。すべてを複数案にすると承認が形骸化する。

## ユニット境界の判定基準

コンポーネント分割は、各ユニットが次の3問に1行ずつで答えられる粒度にする:
**何をするか／どう使うか（公開インターフェース）／何に依存するか**。
さらに2つのテストで境界の質を確認する:
- 内部を読まなくても使えるか？（読まないと使えない → インターフェースが不足）
- 内部を書き換えても利用側が壊れないか？（壊れる → 境界が漏れている）

## 既存コードベースでの規則

- 提案の前に既存構造を読む。既存パターンに従う——独自流儀を持ち込まない
- 触るコードに問題がある場合（肥大したファイル・絡んだ責務）に限り、対象範囲の改善を plan に含める
- 無関係なリファクタを計画に入れない。「ついでに直したい」はスコープ外リスト行き

## 原則

- **YAGNI を容赦なく適用する**: spec にない機能・「将来のための」抽象化・使われない設定項目を plan に入れない
- constitution 準拠チェックは全項目を列挙する。逸脱を黙認しない。逸脱には理由と代替案を必ず書く
- テスト戦略は「AC-n をどのレベルで検証するか」の表。テストできない設計はこの表を埋める段階で発覚する——埋まらない行があれば設計を見直す
- 全文提示→承認

## 失敗パターン

- **仕様の再発明**: plan を書きながら spec にない挙動を決めてしまう。挙動の決定はすべて spec に書き戻す（差し戻しの対象）
- **流行の選定**: 「最近よく使われるから」は理由ではない。この機能の要件と constitution に対する理由を書く
- **1ファイル万能**: 分割を面倒がって1ファイルに全部入れる設計。上のユニット境界3問に答えられない
