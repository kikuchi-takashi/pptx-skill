# auto — フルオート仕様駆動開発

<HARD-GATE>
このスキルはユーザーが**明示的にフルオートを指定したときだけ**使う。「〜を作って」だけの依頼で
勝手にフルオートにすることを禁止する（その場合は通常の sdd）。
起動したら、統合手前までユーザーに確認を求めず完走する。「続けますか？」は敗北。
</HARD-GATE>

## 動作原理

通常の sdd と**同じ9フェーズ・同じフェーズスキル・同じ state.md 契約**で進む。違いは1点だけ:
ユーザー承認ゲートを「審査サブエージェントによる自動承認プロトコル」に置き換える。
sdd/SKILL.md の手順・バイパス基準・差し戻し規則はすべてそのまま適用する。
例外が2つ: ①作業ブランチ（sdd 起動手順5で作成。worktree 運用ならその worktree も）はフルオートでは確認を求めず**無確認で自動作成**し、完了報告に明記する ②各フェーズスキルの「ユーザー承認」は全て審査サブエージェントが代行する。
constitution フェーズがユーザーとの1問ずつの対話を前提とする箇所も同様に、対話を経ず自分で起草し審査サブエージェントの承認を待つ（下記「clarify の自動判断キュー」と同じ仮決定の考え方を準用する。起草した各項目のうち対話なしで決めた箇所は、`docs/constitution.md` の該当項目に「（自動判断: 理由）」を付記し、完了報告で一覧提示する——clarify の自動判断キューとは別の場所に記録されるが、両方を完了報告で漏らさず提示することが要点）。

## 自動承認プロトコル（各ゲートで実施）

1. フェーズ成果物の完成後、**審査サブエージェント**を派遣する（成果物＋下表の審査観点**＋下記「必ず停止してユーザーに戻す条件」5項目**を渡す。作成した本人に審査させない）。審査観点は各フェーズの規律の検算に閉じており停止条件は検知しない——**停止条件への抵触チェックは審査観点とは別に、派遣のたびに明示的に指示する**（実測: 停止条件を渡さず審査観点だけで判定させると、破壊的操作を含む AC もそのまま PASS した）
2. 審査 PASS → state.md に記録して次フェーズへ:
   `- YYYY-MM-DD HH:MM <フェーズ名> 自動承認（auto、審査: <指摘ゼロ等の一言>）`
3. 審査 FAIL → 指摘を修正して再審査。**同一フェーズ3回 FAIL で停止**しユーザーに戻す
4. 審査員のモデルは成果物の複雑さに見合うものを明示指定（フィーチャーレビュー相当は最上位）

| フェーズ | 審査観点（各スキルの規律の検算） |
|---|---|
| constitution | 各項目が機械的に準拠/逸脱を判定できる宣言文か／template.md の分量・構成規則に従っているか |
| specify | AC が全て検証可能か／HOW 混入なし／エッジケース4系統を確認済みか |
| clarify | 曖昧箇所の検出を実施したか（下記「自動判断キュー」参照） |
| plan | constitution 準拠表が全項目埋まっているか／テスト戦略に全 AC があり、検証レベルが template の選定指針に沿うか |
| tasks | 4項目必須・粒度・[P] 妥当性・AC カバレッジ |
| analyze | analyze のレポートに重大ゼロか（重大ありは通常どおり差し戻し） |
| implement | 全タスクの TDD 証拠と検証コマンド出力が揃っているか |
| review | review の判定が「通過」か（重大・重要ゼロ） |
| verify | 全 AC・エッジ・非機能が実行証拠付きで PASS か |
| spec-lite（バイパス時） | 3節（変更内容・受け入れ基準・検証コマンド）が埋まり、バイパス基準3条件を満たすか |

## clarify の自動判断キュー

フルオートでは質問できないため、曖昧箇所は**自問自答で仮決定**し「自動判断キュー」に記録する。
- 仮決定できるのは挙動の細部（clarify「質問の優先順位」でいう挙動・文言レベル）のみ。**データの持ち方・機能の有無のレベルで確信が持てない場合は停止**してユーザーに戻す——ここで間違えると全部が無駄になる
- キューの全項目は spec.md に反映しつつ `clarifications.md` に「自動判断」と明記し、完了報告で一覧提示する（ユーザーが後から覆せるように）

## 収束ループ（ループエンジニアリングの核）

implement → review → verify は一発で通ることを期待しない。FAIL・差し戻しを前提に回す:

```
implement 完了 → review（指摘→修正→再レビュー） → verify（FAIL→implement差し戻し→修正 diff の再タスクレビュー→再verify）
```

- 修正ループの回数を state.md のフェーズ内進捗欄にカウントする
- **ワークフロー全体で修正ループ合計10回**を超えたら停止（収束していない＝仕様か計画に構造的問題がある。ループで殴り続けない）
- 各ループの修正は指摘への対処だけを行う。「ついでの改善」はループを発散させる

## バイパス条件が崩れた場合（バイパス実行中のみ）

sdd のバイパス節が定義する2つのケースを、フルオートでは判断の性質で分けて扱う:

- **条件2だけに抵触する救済ルート**（既存 spec.md の AC 修正が必要になった場合）: AC の期待値変更はユーザーの元依頼の解釈に関わる判断であり、下記停止条件2に該当する。**停止してユーザーに確認**を取ってから続行する
- **3ファイル以上に膨らむ機械的な閾値超過**（sdd のバイパス節「実装中に変更が3ファイル以上に膨らんだら」。**既存テストファイルの更新も1ファイルに数える**——新規追加でなくても対象）: 「どちらが正か」の判断を伴わない機械的な切り替えのため停止しない。テストは資料として保持・実装コードは破棄・spec-lite→spec 昇格・state.md のルート欄を **フル に書き換え**＋差し戻し履歴記録（sdd のバイパス節の記録手順どおり）を自動で実行する。**この実装コード破棄は下記停止条件1の「既存資産の削除」には当たらない**（フルオート実行中に自分が書いたコードであり、事前に存在したユーザー資産ではないため）。切り替えの発生と経緯は完了報告に明記する
- **両方に同時に抵触する複合ケース**（3ファイル超過とAC修正要が同時発生）: 判断を伴う条件2を優先し、**停止してユーザーに確認**する（機械的処理は「判断不要」な場合に限る安全側の原則）

## 必ず停止してユーザーに戻す条件（フルオートでも越えない一線）

1. 破壊的操作が必要になった（既存資産の削除・上書き・force push・外部サービスへの送信。バイパス誤判定によるフルオート自身の実装コード破棄は含まない——上記「バイパス条件が崩れた場合」参照）
2. 仕様の根本的曖昧さ（データの持ち方・機能の有無のレベル）や、実装中に発覚した仕様矛盾で spec または plan の書き直しが必要（どちらが正かの判定はフルオートでは代行しない——ユーザーの元依頼の解釈に関わるため）
3. constitution からの逸脱なしに実装できないと判明した
4. 同一フェーズ審査3回 FAIL、または修正ループ合計10回到達
5. **統合（main へのマージ・PR 作成・push）**——検証が全件 PASS でも、ここから先は必ずユーザーが選ぶ

停止時は「どこまで進んだか（state.md）・何に詰まったか・選択肢」の3点で報告する。state.md は**現在フェーズを進めず、そのフェーズの承認行も追加しない**ことで「停止中」を表す（承認履歴にも差し戻し履歴にも新しい専用の列挙値は増やさない）。停止理由は spec.md の `[要確認]`（仕様内容自体の曖昧さ用）を流用せず、報告本文と差し戻し履歴に直接書く。

## 完了報告（統合手前で必ず提示）

- 検証レポート（N/N PASS、証拠付き）
- 自動承認ログ（state.md 承認履歴の写し）
- 自動判断キューの一覧（clarify の曖昧箇所解消＋constitution の仮決定。覆すなら差し戻せる旨を添える）
- バイパス→フルへの切り替えが発生した場合はその経緯（state.md 差し戻し履歴の写し）
- 修正ループの回数と内訳
- 統合3択（verify の「統合方法の選択肢」と同じ。ユーザーが選ぶ）
