# clarify — 曖昧箇所の解消

## 手順

1. spec.md の `[要確認]` マーカーを集める。加えて、受け入れ基準を読み「2通りに解釈できる箇所」を自分でも探す
2. **1メッセージ1問・選択式優先**で質問する。上限5問。5問を超える曖昧さがある場合は影響の大きい順に5問とし、残りは spec の分割をユーザーに提案する
3. 各回答を `specs/NNN-機能名/clarifications.md` に追記する:

```markdown
## Q1: <質問>
- 回答: <ユーザーの回答>
- 反映先: spec.md <該当節>
```

4. 回答を spec.md 本文へ反映し、対応する `[要確認]` を消す
5. マーカーがゼロになったら、変更差分の要約を提示する

## 質問の優先順位

「影響の大きい順」= 答えによって plan・tasks の形が変わる順。判断基準:
①データの持ち方が変わる質問（最優先） ②機能の有無が変わる質問 ③挙動の細部が変わる質問 ④文言・見た目の質問（最後。実装中でも直せる）。

## 質問の型

**良い質問**（選択式・回答が仕様にそのまま書ける）:

> 重複したタスク名を許しますか？
> A. 許す（同名タスクが複数並ぶ） B. 拒否してエラー表示 C. 自動で連番を付ける（推奨: 一覧での取り違えを防げる）

**悪い質問**:
- 「タスク名の扱いはどうしますか？」——開いた質問は回答が曖昧になり、二度聞きになる
- 「重複チェックは Set と DB ユニーク制約のどちらでやりますか？」——HOW の質問。clarify で聞くのは挙動だけ
- 1メッセージに3問——回答が混ざり、記録が壊れる

推奨案があるなら選択肢に（推奨: 理由）を付ける。ユーザーの判断コストを下げるのも clarify の仕事。

## 回答が新たな曖昧さを生んだ場合

回答を反映した結果、別の箇所が2通りに読めるようになったら、新しい `[要確認]` として spec に記録し、残り質問数の範囲で続けて聞く。上限を超えるなら「解消済み N 件・新規 M 件」を報告して sdd に判断を戻す。

## 自動通過

呼ばれた時点でマーカーがゼロかつ自力検出の曖昧箇所もない場合は、質問せず「曖昧箇所ゼロ」を報告して終了してよい（sdd はこれを state.md に記録する）。**ただし「検出を怠った」と「検出してゼロだった」は別物**。受け入れ基準を1件ずつ読んだ上でゼロと言うこと。
