# SDD スキルセット 実戦トライアル 摩擦ログ

対象: TODO 管理 CLI（追加・一覧・完了、JSON 保存）を SDD フルルート9フェーズ→バイパス1件で通す。

凡例: 迷った / 矛盾 / 不足 / 過剰 / 効いた

## 起動前（sdd 本体の起動手順）

- **効いた**: sdd:起動時の手順 手順2「constitution チェック」— `docs/constitution.md` が無いことを機械的に確認できたので、constitution から開始する判断に迷いがなかった。
- **迷った**: sdd:起動時の手順 / sdd-constitution:手順4 — constitution の承認記録は「作成した機能の state.md」に書く規則だが、state.md は手順5（採番）で初めて作られる。つまり constitution 作成・承認（手順2）→ 採番（手順5）→ state.md に事後記録、という順序になり、手順の番号順だけを読むと「まだ存在しない state.md に記録しろ」と読めて一瞬迷った。実際は state.md 作成後に遡って1件目の承認履歴として書くと解釈した。手順2の説明に「記録は採番後になる」の一言があれば迷わなかった。

## specify フェーズ

- **迷った**: sdd-specify:HOW禁止ルール — ユーザー依頼原文に「データはJSONファイル保存」と明示されている。HOW禁止ルールは「ファイル拡張子」を禁止技術名詞に挙げるが、これは実装者が選んだHOWではなくユーザーが要求したWHAT（永続化形式そのものが要件）。禁止ルールの自己チェックでは「フレームワーク名・ライブラリ名・DB種別・API設計・ファイル構成・クラス名」が対象で「ユーザーが明示的に指定したファイル形式」は例外か区別がない。今回はユーザー要求語彙をそのまま残す（非機能要件の互換性節に留める）解釈にしたが、原則とユーザー入力が衝突するケースの扱いが書かれておらず判断は自己流。
- **効いた**: sdd-specify:エッジケースの探し方（①空・ゼロ②境界③重複・同時④異常系） — この4分類のチェックリストのおかげで「壊れたJSON」「存在しないタスクの完了」「空文字タスク」を機械的に洗い出せた。無ければ見落としていた可能性が高い。
- **不足**: sdd-specify:手順1 — 「6個以上になるなら機能が大きすぎる」の判断基準はストーリー数のみで、AC数やエッジケース数は言及なし。US4個・AC6個で収まったが、AC側の上限が書かれていないため肥大化の歯止めが片方しかない。

## clarify フェーズ

- **効いた**: sdd-clarify:手順1「加えて、受け入れ基準を読み『2通りに解釈できる箇所』を自分でも探す」— spec.md には `[要確認]`（重複タスク名）が1件しかなかったが、この指示に従い読み直したところ「完了操作でタスクをどう指定するか（連番/名前/ID）」が未定義だと判明した。マーカー任せだと1問しか出ず、実運用のバグ元になる箇所を見逃すところだった。
- **効いた**: sdd-clarify:質問の型（選択式＋推奨案） — 「推奨: 〜」を選択肢に付ける形式のおかげで自問自答が速く済み、判断根拠も記録に残った。

## plan フェーズ

- **迷った**: sdd-plan:「大きな選定は複数案で提示する」— 本来は「2〜3案をユーザーに提示してから plan に書く」という2段階（提示→承認→記載）のはずだが、トライアルでは自己承認のため提示と記載を同時に行い、plan.md の表内に代替案を書く形で済ませた。有人運用なら「技術選定の提案メッセージ」という別の成果物（会話上のテキスト）が本来必要で、plan.md に書く内容と全く同じにはならない可能性がある。スキップした分を自覚しておく。
- **不足**: sdd-plan:データモデル節 — テンプレートのコメントは「構造体・スキーマ・ファイル形式。永続化するものは保存場所も」とあるが、保存場所のデフォルト値・CLI引数化の要否は書かれておらず自分で決めた（`--file` オプションとデフォルト `tasks.json`）。CLI固有の「保存先をどう指定させるか」はplanのどの節に書くべきか自明でなかった（データモデルに置いたが、コンポーネント分割の公開インターフェースとも言える）。

## tasks フェーズ

- **迷った**: sdd-tasks:粒度の良い例・悪い例 — 一旦 `[P]` を T1 に付けてから、T2 が T1 に依存する2タスク構成では並列実行できるタスクが存在しないと気づき削除した。「依存のないタスク同士に付ける」という規則自体は明快だが、2タスクしかない小規模機能では「並列できる相手がいない」ケースの扱い（そもそも付けるべきでない）を一度取り違えた。自己レビューで気づけたが、機械的なセルフチェック項目（「[P]を付けたタスクに、実際に並列相手が存在するか」）があれば防げたミス。
- **効いた**: sdd-tasks:タスク境界の基準「独立してテストでき…最小単位」＋「テスト作成と実装は同一タスクにまとめる」— この2つのおかげで「ストアとCLIを分けるがテストは実装に同梱」という自然な2タスク構成に素直に落ち着いた。
- **不足/効いた（trade-off）**: sdd-tasks / sdd-implement:自動選択基準（1〜2ファイル→直接、3ファイル以上→サブエージェント）— 今回の機能規模では2タスクとも変更対象が2ファイルに収まり、サブエージェント派遣の分岐が一度も発生しない構成になった。ミッションが要求する「派遣が必要な場面での代替」を試す機会が自然には来ないため、小粒度な機能では自動選択の分岐ロジックがほぼ検証されない、という点を記録しておく（実装フェーズで詳細を追記）。

## analyze フェーズ

- **効いた**: sdd-analyze:検出例「矛盾: specは即時反映、planは終了時一括書き出し」のような対比パターン — このパターンを手本に spec.md と plan.md を突き合わせたところ、実際に「ファイル未存在時の自動作成」の記述で spec と plan の間に軽微な表現の飛躍があるのを発見できた（AC には影響しないため軽微判定）。検出例が具体的だったから見つけられた。
- **迷った**: sdd-analyze:重大度の較正「迷ったら『放置して実装した場合どのフェーズまで戻るか』で判定」— 見つけた矛盾はどのACのテストにも影響しないため「実装で吸収できる＝軽微」と判定したが、「表現の飛躍」自体は仕様の言葉の精度の問題であり、厳密には spec まで戻る必要がある種類の指摘とも言える。軽微/重大の境界は「テスト可能な挙動への影響」を基準にするのか「文言の正確性」を基準にするのかで結果が変わり、今回は前者を採用した。

## implement フェーズ

- **効いた**: sdd-implement:前提チェック「main/master の場合、作業ブランチの作成を確認」— main で作業を始めていたことに気づき、`001-todo-cli` ブランチを切ってから実装した。この確認がなければ main に直接コミットしていた可能性が高い。
- **不足**: sdd-implement:前提チェック — ブランチ作成のタイミングは明記されているが、「承認済み成果物（spec/plan/tasks/analyze-report/state.md）を作業ブランチとmainのどちらにコミットするか」は書かれていない。今回は作業ブランチ作成後にまとめてコミットしたが、mainに先にコミットしてから分岐する流儀もあり得る。プロジェクトを跨いで再現するには規則化が必要。
- **効いた/過剰（境界事例）**: sdd-implement:サブエージェント派遣の規則「BASE記録」— 本タスクは全タスクが直接実装（1〜2ファイル）だったため、サブエージェント派遣もタスクレビューも発生しなかった。ミッション記載の「派遣できない場合は摩擦ログに記録して直接実装で代替」の状況は、そもそも発生条件（3ファイル以上）を満たすタスクが無かったため一度も訪れなかった。BASE記録はサブエージェント派遣の文脈の規則だが、直接実装でも「タスク単位のコミット」の起点として同じ効能があったため、直接実装でも自主的に記録した。
- **効いた**: sdd-implement:「直接実装のタスクもimplementer-prompt契約に従う（TDD証拠・報告フォーマット）」— この一文のおかげで、直接実装でもRED（ImportErrorで失敗）を都度実行・確認してからGREENに進む規律が保たれた。もしこの一文が無ければ「自分で書くのだから省略してよい」と合理化していたかもしれない。
- **不足**: sdd-implement 全体 — 「本体が直接実装するタスクについて、実装者のステータス契約（DONE/DONE_WITH_CONCERNS/BLOCKED/NEEDS_CONTEXT）を自分自身にどう適用するか」が書かれていない。今回は暗黙に「詰まったら止まる」だけを踏襲したが、直接実装時に自分でBLOCKED相当の判断をどう記録するかの型が定義されていない。

## review フェーズ（フィーチャーレビュー）

- **効いた**: sdd-review:検出例「重複」「重大度の較正」— 実際に diff を読み直したところ、`todo_cli.py` の `add` が `json.JSONDecodeError`（`ValueError`のサブクラス）を暗黙に拾ってしまい、list/doneと異なる生のエラー文を出す実バグを発見した。ミッション指示どおり「実際にdiffを読む」を徹底しなければ見逃していた（自分で書いたコードなので性善説で流していた可能性が高い）。
- **効いた**: sdd-review:差し戻し規則「重大・重要が1件でもあれば implement へ差し戻す。修正後に必ず再レビュー」— この規則どおりに一度「差し戻し」判定を出し、実際に修正・回帰テスト追加・再レビューまで一巡させた。もしこの規則がなければ「軽微に格下げして通過」させていた誘惑があった（自分の実装への甘さ）。
- **不足/迷った**: sdd-review:手順5「レビューはサブエージェントに派遣してよい（推奨）」— 本トライアルでは実装者と同一セッションでレビューも行った（自己レビュー）。「してよい」という任意表現のため規則違反ではないが、実装者本人がレビュアーを兼ねると「自分の書いたロジックは正しいはず」という確証バイアスがかかるリスクは構造的に残る。今回はdiffを1行ずつ読む姿勢で実バグを見つけられたが、これは徹底度に依存しており、スキル側に「本体が実装した機能は特に自己レビューの死角に注意」という明示的な警告があってもよい。

## verify フェーズ

- **効いた**: sdd-verify:禁止フレーズ表＋手順1「ユニットテストの再実行で代用しない」— この指示のおかげで、既に18/18テストPASSしていたにもかかわらず、あえて `todo_cli.py` を実際にサブプロセス起動してAC/エッジケースを再現した。その過程で「ファイル未存在での初回起動時、実際にファイルは作られない（listはsaveしないため）」という実装の実挙動を目視で再確認でき、analyzeで指摘した表現修正が正しかったことを実地で裏取りできた。ユニットテスト結果だけを信じていたら、この「本当にファイルが作られないこと」の実地検証はしなかった可能性が高い。
- **不足**: sdd-verify:手順1 — 検証用に一時ファイル（`_verify_*.json`）をプロジェクトルートに作る必要があったが、「検証時の一時生成物をどこに置き、いつ消すか」の指示がない。今回はコミット直前に手動で `rm` して git 管理外のままにしたが、うっかり `git add -A` 等をすればプロジェクトに混入するリスクがあった。verifyフェーズ専用の作業ディレクトリ（例: `.sdd-verify-tmp/` を.gitignoreする）を規約化すると安全になる。
- **迷った**: sdd-verify:手順7「統合方法の選択肢…①コミットして終了」— 「コミットして終了」が「作業ブランチにコミット済みのまま終了（mainへは統合しない）」なのか「mainへ統合してから終了」なのかが曖昧。実装は各タスク単位で既にコミット済みだったため、①を選んだ場合に何が追加でやることなのか一瞬迷った。トライアルでは「マージまで含めて完了」と解釈し、`git merge --no-ff` でmainへ統合した。フル/バイパスどちらのルートでもこの選択肢がフェーズの最後に出るため、3択それぞれの具体的な操作（例: ①=作業ブランチのままmainにfast-forwardしない/する、②=PR作成コマンド例、③=何もしない）を明記した方がよい。

## バイパスルート（002-todo-list-count）

- **迷った**: sdd:小規模バイパス基準「変更が1〜2ファイルに収まる見込み」— このファイル数にテストファイルを含めるかどうかが明記されていない。今回は実装1ファイル（`todo_cli.py`）＋テスト1ファイル（`test_todo_cli.py`）の計2ファイルとして数え、ぎりぎり基準内と判定した。テストを含めない数え方なら「1ファイル」で余裕を持って判定できたはずで、境界事例では数え方の定義が結果を左右する。
- **不足**: sdd:小規模バイパス基準／sdd-implement — バイパスの成果物（spec-lite.md）をどの `specs/NNN-機能名/` に置くか（既存001に追記するか、新規002を採番するか）の指示がない。今回は「新しい依頼として独立した番号を採番する」と判断したが、逆に「既存機能への軽微な修正だから001配下に置く」という解釈も成り立つ。番号付けの規則（手順5「採番」）はフルルート前提の書き方で、バイパス時の適用が明示されていない。
- **効いた**: sdd-verify:手順1「ユニットテストの再実行で代用しない」— バイパスでも同じ徹底があったため、`合計: 3件` という表示文言が実際にターミナル出力として正しい位置（一覧の最後）に出ることを目視で確認できた。地味だが「N/N」ではなく実際の文字列を見て確認する価値があった。
- **効いた**: sdd-tasks（バイパスでは不使用のはずだが読み直した）:「テスト作成と実装は同一タスクにまとめる」の精神を踏襲し、既存テスト（`test_list_shows_all_tasks_with_status`）の期待値が新しい出力仕様と食い違うことに実装前に気づき、RED確認の一環として修正できた。バイパス手順にはこの「既存テストの期待値更新」への言及がなく、フルルートのTDD規律を流用して埋めた（不足の裏返しとして効いた例）。
- **既存機能への影響確認**: バイパス基準の3条件目「新しい依存関係・データモデル変更を伴わない」は該当なしで即判定できたが、1条件目・2条件目は実際にコードを触ってみるまで確信が持てなかった（結果的に基準内に収まったが、着手前の見積もりの精度は低い）。バイパス判定を「着手前の見込み」だけで固定するのではなく、実装後に基準を再チェックする一言（tasks.mdの「3ファイル以上に膨らんだら」節と同様の仕組み）がspec-lite側にもあるとよい。
